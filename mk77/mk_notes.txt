# integrated_mk — Krish + Myles Integrated Board

This is the integrated firmware for the RP2350 "Proton" board that runs both
Krish's **Blast Gauge** (WS2812 ring + servo) and Myles's **Wave Match**
(pots + TFT) as a single module. It talks to the Celine+Alexia board (see
`../final/`) over UART1 using the `$STRIKE` / `$SOLVED` / `$RESET` protocol.

## Pin assignments

Blast-gauge pins were moved from the standalone version so they don't clash
with Myles's TFT:

| Function              | Pin   | Owner | Notes                            |
|-----------------------|-------|-------|----------------------------------|
| WS2812 DIN            | GP16  | Krish | moved from GP14 (TFT_DC conflict)|
| Servo signal          | GP20  | Krish | moved from GP15 — see note below |
| Start/stop button     | GP5   | Krish | active-low, internal pull-up     |
| TFT CS                | GP17  | Myles | SPI0                              |
| TFT RESET             | GP15  | Myles |                                   |
| TFT DC                | GP14  | Myles |                                   |
| TFT SCK               | GP18  | Myles |                                   |
| TFT MOSI              | GP19  | Myles |                                   |
| RGB LED R / G / B     | 12/11/10 | Myles | common-anode, PWM          |
| Pot (frequency)       | GP41  | Myles | ADC1                              |
| Pot (amplitude)       | GP42  | Myles | ADC2                              |
| Peer UART1 TX         | GP8   | Both  | to peer board's RX                |
| Peer UART1 RX         | GP9   | Both  | to peer board's TX                |
| GND (peer link)       | any GND | Both | **must be common-grounded**       |

> **Servo pin choice — do not "simplify" to GP13.** On the RP2350, PWM slices
> pair adjacent GPIOs. GP12 and GP13 both sit on slice 6 (ch A and ch B)
> which shares one `wrap` register. Myles's LED_R on GP12 wants `wrap = 255`
> for 8-bit brightness; a servo on GP13 would need `wrap = 19999` for a 20 ms
> period — impossible. GP20 lives on slice 2 and is fully independent.

## System FSM

```
        ┌──────────┐  first button OR pot movement
        │ SYS_IDLE │──────────────────────────┐
        └──────────┘                          ▼
                                    ┌──────────────┐
                    all 4 modules   │ SYS_ACTIVE   │─► SYS_DEFUSED
                    solved     ┌────└──────┬───────┘
                               ▼           │ 3 total strikes
                        SYS_DEFUSED        ▼
                                    SYS_EXPLODED
```

Strikes are bomb-wide, not per-module. Any module on either board emitting a
strike increments a shared counter. At 3, the whole bomb explodes.

## Blast gauge internal FSM

```
  BG_IDLE ──press──► BG_ROUND1 ──press+correct──► FLASH_PASS ─► INTER_ROUND ─► BG_ROUND2
                        │                                                         │
                        └──press+wrong──► FLASH_STRIKE ──► INTER_ROUND ──┐       ...
                                                                         ▼
                                                          (same round re-entered)
```

Key difference from the standalone version: **no auto-start**. The first button
press arms the module out of `BG_IDLE`. This lets the player choose which
module to start first, matching Myles's "move a pot to begin" behavior.

The old "3 strikes = exploded" per-module logic was removed — strikes are now
routed up to the system FSM for bomb-wide aggregation.

## UART protocol (UART1 @ 115200 8N1)

All messages are newline-terminated ASCII:

| Message            | Direction     | Meaning                                    |
|--------------------|---------------|--------------------------------------------|
| `$STRIKE:<who>`    | both ways     | module `<who>` on sender's board got a strike |
| `$SOLVED:<module>` | both ways     | `<module>` was just solved                 |
| `$RESET`           | both ways     | reset everything back to IDLE (debug)      |

Module names: `KRISH`, `MYLES` (this board), `CELINE`, `ALEXIA` (peer board).

**Wiring reminder:** cross-connect TX↔RX. This board's GP8 (TX) goes to peer's
RX pin, and this board's GP9 (RX) comes from peer's TX pin. And don't forget
the common ground — serial UART floats without it.

## Building

PlatformIO:
```
cd integrated_mk
pio run
pio run -t upload           # via picoprobe
pio device monitor          # USB serial debug output
```

Plain Pico SDK / CMake: add `add_subdirectory(integrated_mk)` to the top-level
`CMakeLists.txt` (replacing or alongside the existing `add_subdirectory(myles)`).

## Known issues / future work

- **200–300 ms freeze on wave-game strikes**: Myles's `game.c` still has a
  couple of `sleep_ms()` calls inside strike handling. During those, the
  blast gauge servo holds position. Cosmetic, not a correctness issue.
  Fix by replacing with a non-blocking timer in `game_ctx_t`.
- **Morse integration**: once the morse module is working, it comes in on
  this same board (or its own), sharing UART1 at GP8/GP9 with either the
  ESP32 (current) or the peer board. Pick one.
- **No `$RESET` originator yet**: neither board currently broadcasts
  `$RESET`. Wire a physical reset button into one of the boards and have
  it send `$RESET` to the peer.
- **Victory sweep animation** from the standalone blast gauge was dropped
  (it used sleeps). Can be re-added as a tickable animation in `BG_DEFUSED`.
